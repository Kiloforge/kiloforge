# Specification: Fix Unhandled SQLite Write Errors Across All Stores

## Summary

Fix 5 locations where `db.Exec()` return errors are silently discarded, causing data loss without any indication. Agent state, quota tracking, traces, and config migration can all fail silently.

## Context

The SQLite persistence layer has 5 write operations that ignore the error return from `db.Exec()`. This means agent status updates, quota recording, trace storage, and config migration can silently fail — the caller never knows data wasn't persisted.

## Codebase Analysis

- **A6:** `agent_store.go:55-64` — `AddAgent()` ignores Exec error
- **A7:** `agent_store.go:105-112` — `UpdateStatus()` ignores Exec error
- **A8:** `quota_store.go:22-35` — `RecordUsage()` ignores Exec error
- **A9:** `trace_store.go:61-107` — Multiple `Record()` Exec calls ignore errors
- **A10:** `migrate_json.go:68` — `migrateConfig()` ignores Exec error

## Acceptance Criteria

- [ ] All `db.Exec()` calls check and return errors
- [ ] Errors are wrapped with context (fmt.Errorf + %w)
- [ ] Callers of these methods handle the newly-returned errors appropriately
- [ ] Tests verify error propagation from each fixed location
- [ ] No silent data loss possible in the persistence layer

## Dependencies

None

## Out of Scope

- Changing store method signatures to return domain sentinel errors (covered by refactor-error-standardization)
- Adding retry logic for transient SQLite errors

## Technical Notes

For each location: check `_, err := db.Exec(...)`, return `fmt.Errorf("store: operation: %w", err)` on failure. Callers (services, handlers) must handle the new error returns — typically log and return an HTTP 500 or surface to the user.
