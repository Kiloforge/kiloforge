## Summary

Comprehensive E2E tests for the Git origin sync feature: displaying sync status (ahead/behind/synced/diverged), pushing to remote, pulling from remote, and handling sync errors, covering happy path, edge cases, and expected failures.

## Context

Kiloforge bridges local Git repositories with a Gitea instance and optionally with an upstream origin remote. The sync feature allows users to push local changes to the origin remote and pull upstream changes. The relevant endpoints are:

- `POST /api/projects/{slug}/push` — push local commits to origin remote
- `POST /api/projects/{slug}/pull` — pull from origin remote
- `GET /api/projects/{slug}/sync-status` — returns sync state: `synced`, `ahead`, `behind`, or `diverged`, with commit counts

The frontend displays a sync panel per project showing the current status, push/pull buttons, and feedback on operation results. Buttons are disabled when the operation is not applicable (e.g., push disabled when synced).

## Codebase Analysis

### Relevant endpoints

- Pull: `POST /api/projects/{slug}/pull` — returns 200 on success with updated status, or error

### UI components

- Sync status badge: shows colored indicator (green=synced, yellow=ahead, blue=behind, red=diverged)
- Push button: enabled when ahead or diverged, disabled when synced or behind
- Pull button: enabled when behind or diverged, disabled when synced or ahead
- Operation feedback: success/error toast or inline message after push/pull

## Technical Notes

### Test data setup

Tests need projects with configured origin remotes. Use the E2E test server's project seeding to create projects in various sync states. The mock/test Gitea instance (or a bare git repo) can serve as the origin remote for E2E tests.

### Simulating sync states

- **Synced:** project repo and origin are at the same commit
- **Ahead:** add a commit to the local project repo without pushing
- **Behind:** add a commit to the origin repo directly
- **Diverged:** add different commits to both local and origin

Use direct git commands in the test setup to manipulate repo state.

### Button state verification

Use Playwright to check button disabled/enabled state via `isDisabled()` or `toBeDisabled()` matchers. Verify button text or icons reflect the operation.

### Error simulation

- Unreachable remote: configure a project with a non-routable origin URL
- Merge conflict: create diverged state where the same file is modified differently
- Network timeout: may need server-side mock or a slow-responding origin
