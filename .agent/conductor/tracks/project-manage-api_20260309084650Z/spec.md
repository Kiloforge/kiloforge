## Summary

Add REST API endpoints for project registration and removal: `POST /-/api/projects` and `DELETE /-/api/projects/{slug}`. This enables the dashboard (and any API client) to manage projects without the CLI.

## Context

Currently `kf add <remote>` is CLI-only — it clones a repo, creates a Gitea repo, sets up webhooks, and registers the project. There is no way to add or remove projects from the web dashboard. The `GET /-/api/projects` endpoint already exists for listing.

### Current State

| Operation | CLI | API | Dashboard |
|-----------|-----|-----|-----------|
| List projects | `kf status` | `GET /-/api/projects` | Overview page |
| Add project | `kf add <remote>` | - | - |
| Remove project | - | - | - |

### What "add" does (CLI flow to replicate in API)

1. Validate remote URL format (SSH, HTTPS, ssh://)
2. Extract repo name from URL
3. Clone remote to `~/.kiloforge/repos/<slug>/`
4. Create Gitea repo via API
5. Add gitea remote + push main branch
6. Create webhook pointing to orchestrator
7. Create project directory (`~/.kiloforge/projects/<slug>/logs/`)
8. Register in `projects.json`

### What "remove" needs to do

1. Halt all running agents for the project
2. Remove webhook from Gitea repo
3. Optionally delete Gitea repo
4. Remove project from `projects.json`
5. Clean up `~/.kiloforge/projects/<slug>/` directory
6. Optionally clean up `~/.kiloforge/repos/<slug>/` clone

## Technical Notes

- **Schema-first:** Update `openapi.yaml` FIRST, then `make gen-api`, then implement handlers
- **Add endpoint complexity:** The API add is simpler than CLI add — no interactive output, just JSON request/response. The heavy lifting (clone, Gitea setup) runs server-side.
- **Remove safety:** Default remove only deregisters from `projects.json`. The `?cleanup=true` query param triggers filesystem + Gitea cleanup. This prevents accidental data loss.
- **Agent halt on remove:** Must check for running agents for the project and halt them before cleanup.
- **Long-running operation:** Clone + Gitea setup can take seconds. Consider returning 202 Accepted with a status URL, or keep it synchronous since it's local-only (fast clones).
