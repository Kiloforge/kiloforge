## Summary

Add a `GET /api/tracks/{trackId}` endpoint that reads track artifacts from disk (spec.md, plan.md, metadata.json) and returns rich track detail data including specification content, implementation plan, phase/task progress, and metadata.

## Context

Currently the API only exposes minimal track data (id, title, status, project) via `GET /api/tracks`. Rich track artifacts exist on disk in `.agent/conductor/tracks/{trackId}/` — including full specifications, implementation plans, and progress metadata — but none of this is accessible through the API. Users viewing the kanban board or track list have no way to drill into track details.

## Codebase Analysis

- **Track service** (`backend/internal/core/service/track_service.go`): Currently parses `tracks.md` for `TrackEntry{ID, Title, Status}`. Needs a new method to read individual track artifacts from disk.
- **REST handler** (`backend/internal/adapter/rest/api_handler.go`): Has `ListTracks()`, `GenerateTracks()`, `DeleteTrack()` — no detail endpoint. The handler already resolves project paths via `projectManager.GetProject()`.
- **OpenAPI spec** (`backend/api/openapi.yaml`): `Track` schema has only 4 fields. Needs a `TrackDetail` schema with spec, plan, metadata content.
- **Track artifacts on disk**: Each track has `metadata.json` (phases, tasks, type, dates), `spec.md` (full specification), `plan.md` (implementation plan), `index.md` (summary). All are plain text/JSON files.
- **Existing pattern**: `DeleteTrack` already resolves track directories via `filepath.Join(conductorDir, "tracks", trackID)` — the detail endpoint can follow the same pattern.

## Acceptance Criteria

- [ ] `GET /api/tracks/{trackId}?project={slug}` returns full track detail including spec, plan, and metadata
- [ ] Response includes parsed metadata (type, phases, tasks, created, updated) from metadata.json
- [ ] Response includes raw markdown content of spec.md and plan.md
- [ ] Returns 404 if track ID not found or track directory doesn't exist
- [ ] OpenAPI spec updated with `TrackDetail` schema and endpoint definition

## Technical Notes

- Read `metadata.json` as structured JSON; parse phases/tasks counts
- Read `spec.md` and `plan.md` as raw strings — the frontend will render them as markdown
- If `metadata.json` is missing (older tracks), fall back to data from `tracks.md` parsing
- Follow existing `DeleteTrack` pattern for project path resolution and track directory lookup
- The `project` query param is required to locate the correct `.agent/conductor/tracks/` directory
