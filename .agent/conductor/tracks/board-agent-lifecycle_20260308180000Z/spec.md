## Summary

When a Gitea board card is moved backward (e.g., In Progress → Approved), halt the associated developer agent. When re-promoted forward, resume the agent. When an issue is closed without merge (rejected), terminate the agent and clean up the worktree.

## Context

The existing tracks (`impl-board-webhook-sync`) handle forward-flow board sync: tracks move Suggested → Approved → In Progress → In Review → Completed. However, there is no handling for backward movements or rejections. A user dragging a card from "In Progress" back to "Approved" should pause the developer agent, not leave it running on work that's been deprioritized.

- **Issue-to-track mapping:** The `impl-track-board-sync` track creates a `TrackIssue` mapping (track ID → Gitea issue number). This can be reversed to go from issue → track ID.
- **Label-based detection:** `handleIssues()` already receives `label_updated` events (`internal/adapter/rest/server.go:272`). The `impl-board-webhook-sync` track will add status label parsing. This track extends that to detect *backward* label transitions.
- **PR tracking links agents:** `PRTracking` (`internal/core/domain/pr_tracking.go`) links `DeveloperAgentID`, `ReviewerAgentID`, `DeveloperSession` to a track. Can be used to find and halt/resume agents in the review phase.
- **Worktree cleanup exists:** `pool.ReturnByTrackID()` returns a worktree to the pool (`internal/adapter/pool/`).
- **Agent states:** `halted` and `suspended` states exist in the domain model. The `halted` state is appropriate for user-initiated pauses via board.

## Acceptance Criteria

- [ ] Moving a card from "In Progress" to "Approved" or "Suggested" halts the running developer agent
- [ ] Moving a card from "In Review" backward halts both reviewer and developer agents
- [ ] Moving a halted card back to "In Progress" resumes the developer agent
- [ ] Closing an issue without a merged PR terminates the agent and returns the worktree to the pool
- [ ] Adding a `rejected` label terminates the agent and posts a summary comment

- [ ] Agent that cannot be resumed (expired session, missing worktree) is marked `resume-failed` with reason

## Technical Notes

### Column Transition Detection

Status labels follow the pattern `status:<state>` (e.g., `status:in-progress`, `status:approved`). A backward movement is detected when:
- `status:in-progress` is removed AND `status:approved` or `status:suggested` is added
- `status:in-review` is removed AND any earlier-stage label is added

The `label_updated` webhook payload includes the full current label set. Compare against the previous known state (from TrackIssue mapping or agent status) to determine direction.

### Agent Resolution Chain

```
Issue webhook → extract issue number
  → TrackIssue mapping → track ID
  → AgentStore.FindByRef(trackID) → AgentInfo (developer)
  → PRTracking (if exists) → reviewer agent
```

### Halt vs Terminate

- **Halt** (backward move): SIGINT → status `halted`, session preserved, can resume later
- **Terminate** (rejection): SIGINT → status `stopped`, worktree returned to pool, no resume expected
