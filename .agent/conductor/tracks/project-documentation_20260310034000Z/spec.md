## Technical Notes

### Interview Process

The developer implementing this track should operate as an **interviewer** — asking the project creator open-ended questions to elicit the story behind Kiloforge. The interview should cover:

1. **Origin** — What problem were you solving? What was the "aha moment"?
2. **Design philosophy** — Why local-first? Why Gitea? Why agent orchestration?
3. **Architecture decisions** — Why clean architecture? Why the conductor/track system? Why worktrees?
4. **Trade-offs** — What did you consider and reject? What would you do differently?
5. **Vision** — Where is this going? What's the long-term goal?

The agent should ask questions one or two at a time, listen to responses, and synthesize the answers into a polished foreword. The creator's voice should be preserved — first person, conversational, authentic.

### Documentation Structure

```
docs/
  README.md           — Documentation index
  foreword.md         — Author's foreword (interview-derived)
  architecture.md     — High-level architecture overview
  getting-started.md  — Quick start guide
```

### Foreword Format

The foreword should read like a personal essay or letter to the reader. Common patterns in open source:
- **"Why I Built This"** — personal narrative style
- **"A Note from the Author"** — more formal but still personal
- **"Philosophy"** — design principles and motivations

The exact framing should emerge from the interview — the author's natural way of telling the story should guide the format.
