# Specification: Fix Route Pattern Conflict Panic on Startup

**Created:** 2026-03-08T20:01:00Z

## Summary

Fix Go 1.22+ ServeMux panic on startup caused by route pattern conflicts between method-less lock routes and method-prefixed dashboard/Gitea catch-all routes.

## Context

`crelay up` panics immediately with:
```
panic: pattern "GET /" conflicts with pattern "/api/locks/"
```

Go 1.22+ enhanced ServeMux enforces strict pattern precedence. A method-less pattern like `"/-/api/locks/"` implicitly matches all HTTP methods (GET, POST, DELETE, etc.), which conflicts with `"GET /-/"` (the dashboard catch-all) because both match `GET /-/api/locks/...` requests but with different specificity rules.

## Codebase Analysis

- **`lock/handler.go:22-25`**: Routes registered without method prefix:
  ```go
  mux.HandleFunc("/-/api/locks", h.handleList)       // no method
  mux.HandleFunc("/-/api/locks/", h.handleLockAction) // no method
  ```
- **`dashboard/server.go:91`**: Dashboard catch-all with method prefix:
  ```go
  mux.Handle("GET /-/", http.StripPrefix("/-", spaFileServer(...)))
  ```
- **`rest/server.go:196`**: Gitea proxy catch-all without method prefix:
  ```go
  mux.Handle("/", s.giteaProxy)
  ```

The lock handler does its own method dispatch internally (`r.Method == http.MethodPost`, etc.), but the mux sees the method-less registration and flags a conflict.

## Root Cause

Lock handler uses method-less `HandleFunc` registrations while other routes on the same mux use method-prefixed registrations (`"GET /-/..."`). Go 1.22+ mux requires consistent pattern specificity — a method-less pattern and a method-specific pattern cannot both match the same request path.

## Fix

Add explicit method-prefixed route registrations to the lock handler, replacing the single catch-all with specific routes per HTTP method. The lock handler's internal method dispatch should move to route-level method matching.

## Technical Notes

### Approach: Split lock routes by method

Replace:
```go
mux.HandleFunc("/-/api/locks", h.handleList)
mux.HandleFunc("/-/api/locks/", h.handleLockAction)
```

With explicit method-prefixed routes:
```go
mux.HandleFunc("GET /-/api/locks", h.handleList)
mux.HandleFunc("POST /-/api/locks/{scope}/acquire", h.handleAcquire)
mux.HandleFunc("POST /-/api/locks/{scope}/heartbeat", h.handleHeartbeat)
mux.HandleFunc("DELETE /-/api/locks/{scope}", h.handleRelease)
```

This uses Go 1.22+ path parameters (`{scope}`) instead of manual path parsing, and eliminates the method-less catch-all that causes the conflict.

Also ensure the Gitea proxy catch-all doesn't conflict — it may need a `GET /` method prefix or the lock routes need to be registered before it.
