## Summary

Make embedded skills the default installation path. Auto-install required skills during `kf init` from the binary — no `SkillsRepo` config required. The GitHub repo path becomes an optional override for custom/updated skills, not a prerequisite.

## Context

Skills are already embedded in the binary via `//go:embed embedded/*` in `checker.go`. The `InstallEmbedded()` function extracts them. But the system still gates everything on `SkillsRepo` being configured:
- `offerSkillsInstall()` returns early if `SkillsRepo == ""`
- `SkillsBanner` returns null if `status.repo` is falsy
- `/api/skills/update` returns 400 if no repo configured
- Auto-update only runs if `SkillsRepo` is set

This means a fresh `kf init` has no skills, the dashboard shows nothing about skills, and the 412 preflight always fails until the user manually configures a repo.

## Technical Notes

### Init flow change

Replace the optional `offerSkillsInstall()` with a mandatory `installEmbeddedSkills()` that:
1. Calls `ListEmbedded()` to get all available skills
2. For each required skill (`kf-developer`, `kf-reviewer`, `kf-track-generator`), calls `InstallEmbedded(name, skillsDir)`
3. Installs to the default global location (`~/.claude/skills/`)
4. No user prompt needed — these are bundled with the binary
5. Skip if already installed and hash matches (up to date)

### API changes

`GetSkillsStatus`:
- When `SkillsRepo` is empty, still return skill list from `CheckStatus()`
- Set `repo` to `"embedded"` or leave null but still populate `skills` array

`UpdateSkills`:
- When `SkillsRepo` is empty, re-extract from embedded assets instead of returning 400
- This handles the case where skills got corrupted or deleted

### Frontend changes

`SkillsBanner`:
- Remove the `if (!status.repo) return null` guard
- Show status based on `status.skills` array presence instead
