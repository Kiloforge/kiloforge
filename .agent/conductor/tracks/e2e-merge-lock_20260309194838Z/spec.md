## Summary

Comprehensive E2E tests for the merge lock coordination system: acquiring locks with TTL, heartbeat renewal, releasing locks, lock conflict detection, and UI display of lock state, covering happy path, edge cases, and expected failures.

## Context

The merge lock system is a critical coordination primitive in Kiloforge. It prevents concurrent merges by multiple developer agents through an HTTP-based lock API with TTL and heartbeat. The lock endpoints are:

- `POST /api/locks/{scope}/acquire` — acquire a lock with holder name and TTL
- `POST /api/locks/{scope}/heartbeat` — renew lock TTL
- `DELETE /api/locks/{scope}` — release a held lock
- `GET /api/locks` — list all active locks

The frontend displays active locks in the dashboard and receives real-time updates via SSE events (`lock_update`, `lock_released`). These E2E tests verify the full stack — API behavior, SSE delivery, and UI rendering — using the mock agent infrastructure.

## Codebase Analysis

### Relevant endpoints

- Lock acquire: `POST /api/locks/{scope}/acquire` — body: `{"holder": "...", "ttl_seconds": N}`, returns 200 with lock info or 409 on conflict
- Lock heartbeat: `POST /api/locks/{scope}/heartbeat` — body: `{"holder": "..."}`, returns 200 or 404
- Lock release: `DELETE /api/locks/{scope}` — body/header: holder identification, returns 200 or 403/404
- Lock list: `GET /api/locks` — returns array of active locks

### SSE events

- `lock_update` — emitted on acquire and heartbeat, includes lock scope, holder, expiry
- `lock_released` — emitted on release or TTL expiry

### UI components

- Lock list in dashboard shows active locks with holder, scope, remaining TTL
- Real-time updates via SSE subscription

## Acceptance Criteria

- [ ] Acquire merge lock — lock appears in lock list with holder and TTL
- [ ] Heartbeat renews lock TTL — expiry time extends
- [ ] Release lock — lock disappears from list
- [ ] Second acquire attempt while locked returns conflict (409)
- [ ] Lock auto-expires after TTL without heartbeat
- [ ] UI displays active locks with holder, scope, and remaining TTL
- [ ] SSE lock_update/lock_released events update UI in real-time
- [ ] Edge cases: heartbeat after release (no-op), acquire immediately after release
- [ ] Failure cases: release by wrong holder, heartbeat for nonexistent lock, acquire with invalid TTL

## Technical Notes

### Test approach

All tests use the E2E test infrastructure from the infra track. Tests drive both the REST API directly (for setup/assertion) and the Playwright browser (for UI verification).

### Lock TTL testing

For TTL expiry tests, use a short TTL (e.g., 2 seconds) and wait for expiry. Do not rely on wall-clock precision — poll the lock list endpoint until the lock disappears or use a generous timeout.

### SSE verification

Subscribe to the SSE `/events` endpoint before performing lock operations. Capture events in a buffer and assert that the expected events arrive within a timeout window.
