## Summary

Investigate how to gracefully shut down all active Claude Code agents (preserving session IDs), persist their state, and automatically restore them on relay restart. Include error handling for sessions that cannot be restored.

## Context

Crelay spawns multiple CC agents as long-running child processes. Currently:
- Shutting down the relay (ctrl-c, `crelay down`) kills child processes without session preservation
- Session IDs are stored in `state.json` but there's no resume-on-startup logic
- No mechanism to snapshot agent progress (which phase/task they were on)
- No graceful shutdown signal cascade to agents
- Users have no visibility into what was lost on shutdown or what was restored on restart

CC supports `claude --resume <sessionID>` which could enable session recovery, but the limitations and edge cases are unknown.

## Codebase Analysis

- **Agent spawning**: `internal/agent/spawner.go` — `SpawnDeveloper()`, `SpawnReviewer()` use `exec.CommandContext`
- **Session IDs**: Already stored in `AgentInfo.SessionID` (`internal/state/state.go`)
- **Agent halt**: `Store.HaltAgent()` sends SIGINT (`state.go:82-96`)
- **Resume exists**: `Spawner.ResumeDeveloper()` uses `claude --resume <sessionID>` (`spawner.go:106-110`)
- **Status tracking**: `running`, `waiting-review`, `in-review`, `changes-requested`, `approved`, `completed`, `failed`, `stopped`
- **Worktree state**: Pool tracks which worktree is assigned to which agent (`internal/pool/pool.go`)
- **PR tracking**: Links developer→reviewer→PR in `pr-tracking.json`

### What already works
- Session IDs are persisted in state.json
- `ResumeDeveloper()` method exists for review-cycle resumption
- Agent status is tracked and persisted

### What's missing
- No shutdown hook on relay server to gracefully stop all agents
- No resume-on-startup logic
- No snapshot of agent progress beyond status
- No error handling for stale/expired sessions
- No user-facing reporting on restore success/failure
- Unknown: Does CC session expire? What happens resuming after hours/days?

## Technical Notes

### Key questions to answer

1. **Session durability**: How long does a CC session remain resumable? Is there an expiry?
2. **State fidelity**: When resumed, does the agent pick up exactly where it left off, or does it re-read context?
3. **Worktree dependency**: Does resume require the same working directory? What if files changed?
4. **Signal handling**: Does CC handle SIGINT gracefully (save state) or just die?
5. **Concurrent shutdown**: Can we SIGINT N agents simultaneously, or must we sequence?
6. **Resume failures**: What does `claude --resume <expired-session>` output? Exit code?

### Investigation approach

1. Spawn a CC agent, capture session ID, SIGINT it, then `--resume` it
2. Test resume after various delays (seconds, minutes, hours)
3. Test resume with modified working directory
4. Test resume with changed git branch state
5. Document all error messages and exit codes
6. Design state machine for shutdown→persist→restore lifecycle
