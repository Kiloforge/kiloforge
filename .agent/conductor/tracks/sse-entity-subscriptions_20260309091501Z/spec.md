## Technical Notes

### Backend Event Emission Strategy

Two approaches based on state source:

1. **Mutation-driven** (preferred for service-owned state): Emit events directly in service/handler methods after successful mutations.
   - Board: `MoveCard()` → emit `board_update`
   - Locks: `Acquire()`/`Release()` → emit `lock_update`/`lock_released`
   - Projects: `Add()`/`Remove()` → emit `project_update`/`project_removed`

2. **Watcher-driven** (required for file-based state): Extend the existing watcher to poll and delta-detect.
   - Tracks: discovered from disk (`.agent/conductor/tracks/`), no mutation API — watcher must poll
   - Traces: in-memory store, but spans arrive via OTLP receiver — watcher or store hook

### Frontend Pattern

Each hook follows the same pattern used by `useAgents`:
1. Initial REST fetch for full state
2. SSE handler merges incremental updates into state
3. Remove `setInterval` polling
4. Optional: keep a long-interval (e.g., 5min) background sync as drift protection

### Watcher Extension

The existing watcher's `checkAndBroadcast` adds track and trace delta detection alongside agents/quota. The `watcherState` struct gains `tracks map[string]string` (id→status) and `traceCount int` fields.
