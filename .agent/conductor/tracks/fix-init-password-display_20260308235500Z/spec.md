# Specification: Fix Init Password Display

**Track ID:** fix-init-password-display_20260308235500Z
**Type:** Bug
**Created:** 2026-03-08T23:55:00Z
**Status:** Draft

## Summary

Fix `kf init` so the Gitea admin password is always displayed to the user when it was auto-generated. Currently on re-init (Gitea already running), the idempotency early-exit path shows no password. On first init, the password IS displayed but the user sees it only once — if they miss it, there's no way to recover it.

## Context

The `remove-password-persistence` track correctly removed the password from `config.json` persistence. However, this created two problems:

1. **Re-init shows blank password** — The idempotency check at `init.go:78-83` exits early when Gitea is already running, printing only URL and Data — no password. Since the password isn't in config, it can't be recovered.

2. **Password only shown once** — On first init, the password is generated and displayed (line 141), but since it's not persisted, the user can never see it again.

## Codebase Analysis

- **`cli/init.go:71-73`** — generates random password if not provided via flag/env/config
- **`cli/init.go:78-83`** — idempotency early exit: `"Gitea is already running."` with URL and Data only, no password
- **`cli/init.go:134`** — `cfg.Save()` uses `json:"gitea_admin_pass,omitempty"` — empty field not written. But the field IS populated at save time, so the password IS being saved despite the intent of the remove-password track
- **`cli/init.go:141`** — `fmt.Printf("  Admin:      %s / %s\n", cfg.GiteaAdminUser, cfg.GiteaAdminPass)` — displays password on first init success path

## Root Cause

Two interacting issues:
1. The password IS actually being persisted to config.json (the `omitempty` tag only omits if empty, and it's not empty at save time). This contradicts the intent of the password-removal track.
2. On re-init with Gitea running, the password resolves from saved config but the early-exit path doesn't display it.

## Acceptance Criteria

- [ ] Password is NOT persisted to `config.json` (honor the remove-password-persistence intent)
- [ ] On first init: password is displayed to the user in the success output
- [ ] On re-init (Gitea already running): password is NOT displayed (it's not known), but a helpful message tells the user how to reset it or where to find it
- [ ] If password was provided via `--admin-pass` flag or `KF_GITEA_ADMIN_PASS` env var: user already knows it, no special handling needed
- [ ] `kf init --admin-pass <pass>` works for setting a known password

## Dependencies

- None

## Out of Scope

- Password reset functionality
- Changing how Gitea authenticates (API token is used for programmatic access)

## Technical Notes

**Fix approach:**

1. Clear `GiteaAdminPass` before saving config:
```go
pass := cfg.GiteaAdminPass  // save for display
cfg.GiteaAdminPass = ""     // don't persist
if err := cfg.Save(); err != nil { ... }
cfg.GiteaAdminPass = pass   // restore for display
```

2. On re-init early exit, note that password is not stored:
```go
fmt.Println("Gitea is already running.")
fmt.Printf("  URL:  %s\n", cfg.GiteaURL())
fmt.Printf("  Data: %s\n", cfg.DataDir)
if cfg.GiteaAdminPass != "" {
    fmt.Printf("  Admin: %s / %s\n", cfg.GiteaAdminUser, cfg.GiteaAdminPass)
} else {
    fmt.Printf("  Admin: %s (password not stored — use --admin-pass to set a known one)\n", cfg.GiteaAdminUser)
}
```
