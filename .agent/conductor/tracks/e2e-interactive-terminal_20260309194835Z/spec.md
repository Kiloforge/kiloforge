## Summary

Comprehensive E2E tests for the interactive agent terminal: spawning interactive agents, WebSocket connection, sending input, receiving output, buffer replay on reconnect, and multi-client behavior, covering happy path, edge cases, and expected failures.

## Context

Kiloforge supports interactive agents that communicate via WebSocket. When a user spawns an interactive agent, the dashboard opens a terminal UI that connects over WebSocket to the backend. The WebSocket protocol uses typed messages: `input` (client to server — stdin), `output` (server to client — stdout), `status` (agent status changes), and `error` (error messages). The mock agent in interactive mode reads stdin and echoes content_block_delta events to stdout.

The terminal UI must handle reconnection gracefully: on reconnect, the server replays the output buffer (up to 500 lines) so the user sees previous output. Multiple browser tabs can observe the same terminal, but only the primary client can send input.

## Codebase Analysis

### WebSocket handler (`backend/internal/adapter/rest/`)

The WebSocket endpoint at `/api/agents/{agentId}/ws` upgrades to WebSocket and:
- Streams agent stdout as `output` messages
- Forwards `input` messages to agent stdin
- Sends `status` messages on agent state changes
- Sends `error` messages for agent errors
- Maintains a circular buffer of recent output (500 lines) for replay on reconnect

### Frontend terminal component

The React terminal component connects to the WebSocket endpoint, renders output in a terminal-like UI, and captures keyboard input to send as `input` messages.

### Mock agent interactive mode

When `MOCK_AGENT_INTERACTIVE=true`, the mock agent:
1. Emits an `init` event
2. Reads lines from stdin
3. For each line, emits a `content_block_delta` echoing the input
4. On EOF or empty line, emits `result` and exits

### WebSocket message types

```json
// Client -> Server (input)
{"type": "input", "data": "user typed text\n"}

// Server -> Client (output)
{"type": "output", "data": "Hello from mock agent"}

// Server -> Client (status)
{"type": "status", "status": "running"}

// Server -> Client (error)
{"type": "error", "message": "agent process exited with code 1"}
```

### Multi-client testing with Playwright

Use Playwright's `browser.newContext()` to create separate browser contexts simulating multiple clients. The first connection is the primary client (can send input), subsequent connections are observers (receive output only).

### Buffer replay testing

1. Spawn interactive agent
2. Send several inputs, accumulate output
3. Close WebSocket connection
4. Reconnect — verify all previous output lines are replayed in order
5. Verify new output continues streaming after replay
