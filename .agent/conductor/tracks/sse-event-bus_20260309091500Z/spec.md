## Summary

Refactor the SSEHub from a dashboard-internal broadcast mechanism into a shared, injectable event bus that any service or handler can emit events to, with typed event definitions and comprehensive tests.

## Context

The current SSE infrastructure (`dashboard/sse.go`) is tightly coupled to the dashboard package. Only the background watcher (`dashboard/watcher.go`) can emit events, and it does so by polling agent/quota state every 2 seconds. This polling approach cannot scale to additional entity types — each new entity would add more disk reads and comparison logic to the monolithic `checkAndBroadcast` function.

To support real-time updates for all entity types (tracks, board, traces, projects, locks), the event bus must be:
1. **Shared** — accessible from REST handlers, services, and the dashboard
2. **Injectable** — passed as a port/interface, not a concrete dependency
3. **Typed** — events use well-defined types, not `any`
4. **Testable** — event emission can be verified in unit tests

## Technical Notes

### Architecture

1. **Port interface** (`core/port/event_bus.go`):
   ```go
   type Event struct {
       Type string
       Data any
   }

   type EventBus interface {
       Publish(event Event)
       Subscribe() <-chan Event
       Unsubscribe(ch <-chan Event)
       ClientCount() int
   }
   ```

2. **SSEHub adapter** — the existing `dashboard.SSEHub` gets refactored to implement `port.EventBus`. It moves conceptually from being dashboard-internal to being a shared adapter.

3. **Typed event constructors** — helper functions like `NewTrackUpdateEvent(track)`, `NewBoardUpdateEvent(card)` ensure consistent event type strings and payload shapes.

4. **Watcher stays** — the background watcher remains for file-based state (agents are stored as JSON files on disk with no mutation hooks). It simply uses the event bus interface instead of direct hub access.

5. **DI wiring** — the event bus is created once at startup and injected into both the dashboard server and the REST API handler via their option structs.

### What stays the same
- SSE HTTP endpoint (`/-/events`) and its handler
- Frontend `useSSE` hook (already handles dynamic event types)
- Non-blocking broadcast with slow-client drop semantics
