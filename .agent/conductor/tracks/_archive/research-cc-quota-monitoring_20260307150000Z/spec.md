## Summary

Investigate approaches for monitoring Claude Code quota usage across concurrent agent workers and handling quota exhaustion gracefully (backoff, queuing, user notification).

## Context

Crelay spawns multiple Claude Code agents as child processes (`claude -p ... --session-id ... --output-format stream-json`). Each agent consumes API quota independently. Currently there is no visibility into quota consumption, no detection of rate limiting, and no graceful handling when quota is exhausted. Workers simply fail or hang.

## Codebase Analysis

- Agents spawned via `exec.CommandContext` in `internal/agent/spawner.go`
- Agent stdout is piped and monitored in a goroutine (spawner.go:87-101)
- `--output-format stream-json` means structured output is available for parsing
- No existing error handling for 429/quota errors
- State tracked in `internal/state/state.go` — agent status can be `running`, `failed`, `completed`, etc.

### Claude Code quota mechanisms discovered

- **OpenTelemetry export**: `CLAUDE_CODE_ENABLE_TELEMETRY=1` enables metrics (`claude_code.token.usage`, `claude_code.cost.usage`, `claude_code.api_error`)
- **API rate limit headers**: `anthropic-ratelimit-*-remaining` on every response (not directly accessible from CC CLI)
- **Error codes**: HTTP 429 (rate limit), 529 (overloaded), with `retry-after` header
- **No direct quota-checking API** — must be inferred from response headers, OTel events, or error output
- **CC behavior on quota hit**: Unknown — needs investigation (does it retry? wait? crash?)
