# Specification: Fix Gitea Push Authentication — Embed Credentials in Remote URL

**Created:** 2026-03-09T22:40:00Z

## Summary

`kf add` fails with "could not read Username for 'http://localhost:4000': Device not configured" because the Gitea remote URL has no credentials embedded. Git tries to prompt for a username/password but there's no TTY. Fix by embedding `user:pass@` in the HTTP remote URL.

## Context

The Gitea remote URL is built as `http://localhost:4000/kiloforger/repo.git` (line 162 of `add.go`). When `git push` runs, git needs credentials for the HTTP endpoint but can't prompt interactively. The admin user and password are available in the config (`cfg.GiteaAdminUser`, `cfg.GiteaAdminPass`) — they just need to be embedded in the URL.

Note: `GiteaAdminPass` is stripped from the saved config JSON (`json_adapter.go:42`), so it's only available in-memory during the session that ran `kf init` (which generates the password). For subsequent sessions, the password needs to come from env (`KF_GITEA_ADMIN_PASS`) or be re-read from where init stored it. However, since the `gitea-proxy-authn` track implemented reverse proxy authentication, we should also check if API token auth can be used instead of password auth for git push.

## Codebase Analysis

### Current remote URL construction

```go
// add.go:162
giteaRemoteURL := fmt.Sprintf("%s/%s/%s.git", cfg.GiteaURL(), cfg.GiteaAdminUser, repoName)
// Result: http://localhost:4000/kiloforger/repo.git
```

### Available credentials

- `cfg.GiteaAdminUser` — always available (default: "kiloforger")
- `cfg.GiteaAdminPass` — available in-memory during init session, stripped from saved JSON
- `cfg.APIToken` — Gitea API token, persisted in config JSON

### Fix options

1. **Embed user:pass in URL** — `http://kiloforger:password@localhost:4000/kiloforger/repo.git`
   - Pro: Simple, works immediately
   - Con: Password may not be available after restart (stripped from config)

2. **Use API token as password** — Gitea supports `http://user:token@host/repo.git` with the API token as the password
   - Pro: Token is always persisted in config
   - Con: Need to verify Gitea supports this for git push

3. **Use git credential helper** — configure `git -c credential.helper='!echo password=...'`
   - Pro: Doesn't embed creds in remote URL
   - Con: More complex

**Recommended: Option 2** — use the API token as the password. Gitea supports `http://user:token@host` for git operations. The token is always available since it's persisted in config JSON.

## Technical Notes

### Fix

```go
// Use API token as password for git HTTP auth
giteaRemoteURL := fmt.Sprintf("http://%s:%s@localhost:%d/%s/%s.git",
    cfg.GiteaAdminUser,
    url.PathEscape(cfg.APIToken),  // token as password
    cfg.GiteaPort,
    cfg.GiteaAdminUser,
    repoName,
)
```
