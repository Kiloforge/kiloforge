## Summary

Consolidate the relay server (:3001) and dashboard server (:3002) into a single HTTP server, and add a reverse proxy to Gitea (:3000). Users access one host — all API, UI, and git forge routes served from a single port.

## Context

Currently kiloforge exposes three separate ports:
- **:3000** — Gitea (Docker, git forge UI)
- **:3001** — Relay (webhooks, health, lock API)
- **:3002** — Dashboard (agent monitoring, quota, SSE, static UI)

This is confusing — users must remember multiple ports. Consolidating into a single server with path-based routing provides one URL for everything.

## Codebase Analysis

- **Relay server**: `internal/adapter/rest/server.go` — HTTP mux with `/webhook`, `/health`, `/api/locks/*`
- **Dashboard server**: `internal/adapter/dashboard/server.go` — HTTP mux with `/api/agents`, `/api/quota`, `/api/tracks`, `/api/status`, `/events`, `/` (static)
- **CLI up.go**: `internal/adapter/cli/up.go` — starts Gitea, then dashboard goroutine, then relay (blocking)
- **Config**: `DashboardPort` (3002), `RelayPort` (3001), `GiteaPort` (3000) — all configurable
- Dashboard and relay are independent HTTP servers with no shared mux

## Technical Notes

### Reverse proxy approach

```go
giteaProxy := httputil.NewSingleHostReverseProxy(giteaURL)
mux.Handle("/gitea/", http.StripPrefix("/gitea", giteaProxy))
```

Gitea supports `ROOT_URL` and `[server].ROOT_URL` config for sub-path hosting — set to `http://localhost:3001/gitea/` in docker-compose env.

### Unified server assembly

The relay server's `Run()` method mounts dashboard routes and gitea proxy onto its mux before starting. Dashboard becomes a route registrar rather than its own server.
