# Specification: Extend Gitea Client with Issue, Label, and Project Board APIs

## Summary

Add Gitea API methods for creating/updating issues, managing labels, and project board (kanban) operations — columns, cards, and card movement. Foundation for track-to-board sync.

## Context

The Gitea client currently supports 14 methods covering repos, PRs, webhooks, and SSH keys. To sync conductor tracks with a Gitea kanban board, we need issue CRUD, label management, and project board APIs. Gitea's REST API v1 supports all of these.

## Codebase Analysis

- **Gitea client**: `internal/adapter/gitea/client.go` — 14 methods, generic `do()` HTTP helper, token+basic auth
- **Port interface**: `internal/core/port/gitea_client.go` — subset of client methods (PR-focused)
- **Tests**: `internal/adapter/gitea/client_test.go` — comprehensive httptest-based table-driven tests
- **Pattern**: All API calls use `c.do(ctx, method, path, body)` → `([]byte, error)`
- **Label creation**: `AddLabel()` already creates labels if missing + adds to issue — good pattern to follow

### API endpoints to implement

```
POST   /api/v1/repos/{owner}/{repo}/issues                        — create issue
PATCH  /api/v1/repos/{owner}/{repo}/issues/{number}               — update issue
GET    /api/v1/repos/{owner}/{repo}/issues?state=X&labels=Y       — list issues
POST   /api/v1/repos/{owner}/{repo}/labels                        — create label (already partial)
POST   /api/v1/repos/{owner}/{repo}/projects                      — create project board
GET    /api/v1/repos/{owner}/{repo}/projects                      — list projects
POST   /api/v1/repos/{owner}/{repo}/projects/{id}/columns         — create column
GET    /api/v1/repos/{owner}/{repo}/projects/{id}/columns         — list columns
POST   /api/v1/repos/{owner}/{repo}/projects/{id}/columns/{id}/cards — create card
PATCH  /api/v1/repos/{owner}/{repo}/projects/{id}/columns/{id}/cards/{id} — move card
```

### Issue types for return values

```go
type Issue struct {
    Number int    `json:"number"`
    Title  string `json:"title"`
    State  string `json:"state"` // "open", "closed"
}

type LabelDef struct {
    Name  string `json:"name"`
    Color string `json:"color"`
}

type Project struct {
    ID    int    `json:"id"`
    Title string `json:"title"`
}

type Column struct {
    ID    int    `json:"id"`
    Title string `json:"title"`
}
```

These can live in the gitea adapter package as API response types (not domain types).
