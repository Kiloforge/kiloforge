# Specification: Handle 412 Skills Missing with Install Prompt Instead of Error

**Track ID:** fix-skills-412-prompt_20260309233000Z
**Type:** Bug
**Created:** 2026-03-09T23:30:00Z
**Status:** Draft

## Summary

When spawning an interactive agent or generating tracks, a 412 "skills not installed" response shows as a raw error toast. Instead, the frontend should intercept 412 and prompt the user to install skills (using the existing skills update mechanism), then retry the action.

## Context

The backend pre-flight check validates that required skills are installed before spawning agents. If skills are missing, it returns 412 with a `SkillsMissingResponse` containing the missing skill names. The frontend doesn't handle 412 specially — it falls through to the global error toast, showing a confusing raw error message.

The existing `SkillsBanner` component and `useSkillsStatus` hook already have the "Install Skills" UI and `triggerUpdate()` API. The fix should reuse this mechanism when a 412 is received.

### Desired flow

1. User clicks "Generate Tracks" or spawns interactive agent
2. Backend returns 412 (skills missing)
3. Frontend intercepts 412 — shows a dialog/prompt: "Required skills not installed. Install now?"
4. User clicks "Install" → calls `triggerUpdate(true)` → waits for completion
5. On success, automatically retries the original action
6. If install fails, shows error

## Technical Notes

The pattern should mirror how 403 (consent) is handled:

```tsx
onError: (err) => {
  if (err instanceof FetchError && err.status === 403) {
    consent.requestConsent(() => retryAction());
  } else if (err instanceof FetchError && err.status === 412) {
    // Show skills install prompt, retry on success
    promptSkillsInstall(() => retryAction());
  }
},
```

Consider creating a `useSkillsPrompt` hook or extending `useSkillsStatus` with a `requestInstall(onComplete)` pattern similar to how consent works with `requestConsent(callback)`.
