## Summary

Comprehensive E2E tests for the quota and token usage display: stat cards showing token counts and costs, rate limit indicators, quota updates via SSE, and cost estimation accuracy, covering happy path, edge cases, and expected failures.

## Context

Kiloforge tracks token usage and costs across all spawned agents. The quota endpoint returns aggregated token counts (input, output, cache), estimated costs in USD, and rate limit status. The frontend renders stat cards for each metric and a rate limit badge that appears when the API is rate-limited.

The relevant endpoint is:

- `GET /api/quota` — returns token counts, cost estimates, and rate limit status

Real-time updates arrive via `quota_update` SSE events as agents consume tokens.

## Codebase Analysis

### Relevant endpoints

- Quota: `GET /api/quota` — returns:
  ```json
  {

    "cache_write_tokens": N,
  ```

### SSE events

- `quota_update` — emitted when agent usage is recorded, includes updated quota fields

### UI components

- Stat cards: individual cards for input tokens, output tokens, cache tokens, and total cost
- Cost display: formatted as USD (e.g., "$1.23")
- Rate limit badge: appears when `rate_limited` is true, shows countdown timer based on `rate_limit_retry_after`
- Rate limit badge hides when rate limiting clears

## Technical Notes

### Test data seeding

Seed quota data directly via the test server's service layer or by spawning mock agents that report known token counts. The mock agent's `result` event includes `usage` (input_tokens, output_tokens) and `cost` fields that the server aggregates.

### Cost formatting verification

Test various cost values:
- Zero: `$0.00`
- Small: `$0.0012` (verify precision)
- Normal: `$1.23`
- Large: `$1,234.56` (verify comma formatting)

### Rate limit testing

Set `rate_limited: true` and `rate_limit_retry_after: 10` in seeded data. Verify:
- Badge appears with countdown starting at 10
- Countdown decrements (or at least badge text updates)
- Badge disappears when rate limiting clears (send SSE with `rate_limited: false`)

### SSE coalescing

When rapid `quota_update` events arrive (e.g., 10 per second), verify the UI does not flicker or show intermediate states. The final displayed values should match the last event received.
